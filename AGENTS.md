# AGENTS.md

このファイルは、CodexなどのAIコーディングエージェントがこのリポジトリを編集するときの作業方針だけを書く場所です。
教材本文、学習ルート、章立て、到達目標はここに重複して書かないでください。

## 正本

教材内容と学習ルートの正本は次のファイルです。

- `docs/index.md`
- `docs/route/index.md`

READMEやこのファイルと正本の内容がずれた場合は、正本を優先してください。
部、章、ファイル一覧、到達目標、学習ルートの具体的な数を `README.md` や `AGENTS.md` に固定しないでください。

## シラバスと各部indexの関係

`docs/route/index.md` は、学習ルート全体のシラバスです。
各部の目的、学ぶ範囲、到達目標を高めの抽象度で管理します。

各部の `index.md` は、その部の入口ページ兼、本文作成の設計図です。
その部で扱う具体的な章立て、回収する操作、確認コマンド、注意点は各部の `index.md` に書きます。

編集時は、次の役割分担を守ってください。

- シラバスには、細かいコマンド一覧や章ごとの本文設計を増やしすぎない。
- 各部indexには、その部を実際に作るための具体的な章立てと回収対象を書く。
- 各章本文には、学習者がその場で読む説明、手順、確認方法を書く。
- 部構成や到達目標を変えるときは、まずシラバスを更新し、その後で該当する各部indexを具体化する。
- 章の追加、削除、順序変更をしたときは、該当する各部indexとサイドバーも確認する。
- シラバス、各部index、章本文で同じ説明を長く重複させない。抽象度を変えて役割を分ける。

## ドキュメント追加の進め方

新しい部や章を追加するときは、いきなり本文を大量に書かないでください。
まず、その部で何を回収する必要があるかを各部indexで確認します。

特に次を確認してください。

- 以前の部で、意味を十分に説明しないまま実行させた操作や出した用語が回収できているか。
- 初学者にとって完全初出の概念を、前提知識のように扱っていないか。
- その部で初めて出す概念は、本文中で短く説明するか、リファレンスへ逃がすか、後の部で扱うことを明示しているか。
- 第0部で導入を優先したツール、コマンド、設定変更の意味が、第1部以降で順番に回収される構成になっているか。

新しい部を作るときは、次の順番で進めてください。

1. シラバスで、その部の目的、学ぶ範囲、到達目標を確認する。
2. 各部indexで、その部の具体的な回収対象、章立て案、確認ポイントを作り込む。
3. 章立てを決める。
4. 最小構成で章ファイルを作り、各章に見出し、ゴール、扱うこと、次章リンクだけを入れる。
5. サイドバーに章を並べ、Docusaurusビルドでリンク切れを確認する。
6. 本文は一気に全部書かず、数章ずつ品質を落とさずに作り込む。
7. まとまった構成変更や本文追加ごとに作業を区切る。

章本文を書くときは、次の点を守ってください。

- その章で初めて出る用語を放置しない。
- 前の章で扱っていない内容を、学習者が知っている前提で使わない。
- コマンドを出す場合は、何をするか、なぜ必要か、何が起きると困るか、どう確認するかを説明する。
- AIに相談する例は、その時点で学習者がClone前なのかClone後なのかを推論してから書く。
- 本文が重くなりすぎる背景知識は `docs/reference/` に分離する。

### AIに練習問題を出してもらう場合

教材内でAIに練習問題やクイズを出してもらうプロンプトを書く場合は、学習者が答えやすい形式にしてください。

- 各部には、理解確認のための練習問題を少なくとも2か所程度入れることを基本にする。
- 練習問題は、概念の比較、危険な操作の見分け、AIへの頼み方、作業後の確認など、判断が必要な箇所に置く。
- 問題数は、本文中の軽い練習なら5問程度を基本にする。
- 一問一答形式にする。
- AIには、1問ずつ問題を表示し、学習者の回答を待つように指示する。
- 選択式にできる内容は、A/B/Cなどの選択肢を使う。
- 選択肢の意味は、各問題の直下に毎回表示させる。
- 学習者の回答前に、答え、採点、解説を表示しないように指示する。
- 学習者が回答したあとに、その問題だけを採点し、理由を解説させる。
- 解説後に、次の問題を1問だけ出すように指示する。
- コマンド練習では、AIにコマンドを実行させないことを明示する。

### 章本文の理想形

第0部全体と第1部の1〜4章は、この教材で求める説明の粒度、順序、画像の使い方の基準です。
第2部以降や未整備の章を直すときは、まずこれらの章を正解例として読み、同じ温度感に寄せてください。

章本文は、単に情報を並べるのではなく、学習者がその場で判断できる順番にします。
基本形は次の流れです。

1. この章でできるようになることを、行動や判断の形で書く。
2. 操作の前に、まず知っておくこと、今は理解を保留してよいこと、危ない点を短く示す。
3. 必要なら、概念図や比較図で頭の中のモデルを先に作る。
4. 手順は小さく区切り、1つずつ実行できる形にする。
5. コマンドや画面操作の直後に、何をしたのか、どこに影響したのか、何で確認するのかを書く。
6. エラーや分岐がある場合は、次へ進む条件と止まる条件を明示する。
7. 操作後に「何が起きたのか」で、実行したことを分類し直す。
8. 必要に応じて「運用者の視点」で、実務上の判断、AIへの頼み方、安全確認に接続する。
9. `AIに聞いてみよう` は、その時点で学習者が本当に使えるAIと見せられる情報に合わせる。
10. 最後に次章へのつながりを書く。

説明は、初心者向けに薄くするのではなく、未説明の操作を放置しないために丁寧に分解します。
ただし、その章の本題から外れる背景知識は深追いせず、「後の章で回収する」「リファレンスで確認する」と明示して逃がします。
第0部のように導入を優先する章では、意味を完全に説明するよりも、公式手順か、実行後に確認できるか、失敗したら止まれるかを優先します。
第1部以降の回収章では、第0部で触れた操作を「何をしたか」「どこに影響したか」「どう確認するか」「AIにどう相談するか」に分解します。

## 編集方針

- 学習者向けの文章は日本語で書く。
- Vibe Codingを「AIに丸投げすること」として扱わない。
- 人間が目的、判断、責任を持ち、AIを相棒として使う教材であることを保つ。
- `README.md` は学習者向けの入口にとどめ、詳しい説明は `docs/` に置く。
- 詳しい補足や寄り道は、必要に応じて `docs/reference/` に置く。
- 初心者向けの一本道を壊さない。
- 未説明の専門用語を避ける。英語の技術用語を使う場合は、日本語で意味を補う。
- コピー用コマンドをMarkdownテーブルに入れない。
- コピー用コマンドの先頭に `$` を付けない。
- VS Code、branch / Pull Request運用、asdf、Nix、devenv、tmux、AWS、VercelなどをMVPの必須手順にしない。
- 教材リポジトリと、学習者が作る成果物リポジトリを混同させない。

## 「AIに聞いてみよう」の作り方

教材内に `AIに聞いてみよう` を書くときは、学習者がその時点でどのAIに何を見せられる状態なのかを先に推論してください。
その状況で本当に成立する聞き方だけを載せます。

### Clone前

Clone前の学習者は、まだこのリポジトリをローカルに持っていません。
CodexやClaude Codeも使えない場合があります。
この段階で想定する相手は、Web版のChatGPT、Claude、Geminiなどです。

Clone前のプロンプト例では、次を守ってください。

- AIが教材本文やリポジトリを見られない前提で書く。
- 「この教材では」「第0部では」だけに依存した聞き方をしない。
- 学習者の目的、OS、実行したコマンド、出たエラー、今いるディレクトリを明示させる。
- パスワード、APIキー、トークン、秘密鍵、ログイン認証コードを貼らないことを必ず書く。
- 「何を確認すればよいか」「次に見るべき場所はどこか」のように、次の行動が返る聞き方にする。

### Clone後

Clone後の学習者は、ローカルに教材リポジトリを持っています。
CodexまたはClaude Codeをリポジトリ内で起動できる場合があります。
この段階では、AIがローカルファイルやGit状態を見られる可能性があります。

Clone後のプロンプト例では、次を守ってください。

- AIをどのディレクトリで起動しているかを確認させる。
- まずは `pwd`、`git status`、主要ファイルの確認など、読み取り中心の依頼にする。
- 学習者が許可するまで、ファイル編集、commit、push、削除、インストールをさせない。
- 「このリポジトリを読んで」と依頼する場合でも、見るべきファイルを具体的に示す。
- 教材リポジトリと成果物リポジトリを混同しないように書く。
- 変更を伴う依頼では、実行前に変更予定ファイルと理由を説明させる。

## 安全方針

学習者の環境に影響するコマンドを説明するときは、次を確認できるように書いてください。

1. 何をするコマンドか
2. なぜ必要か
3. 何が起きると困るか
4. 結果をどう確認するか
5. 可能なら、どう戻すか

特に次を扱うときは慎重に説明してください。

- `sudo`
- `rm`
- `chmod`
- `chown`
- `curl | sh`
- PATH変更
- `.zshrc` 編集
- `chsh`
- GitHubへのpush
- GitHub Pagesでの公開
- `.env`、シークレット、トークン、APIキー

## 画像方針

- 学習者向けの説明画像やインフォグラフィックを新しく作る場合は `imagegen` を使う。
- 説明図やインフォグラフィックを、AIエージェントが独自にSVGで作ることは禁止する。
- SVGを作成・編集するのは、ユーザーが明示的にSVGを求めた場合だけにする。
- コマンド文字列など正確性が必要な図でも、まず `imagegen` で作り、文字の崩れや誤字を目視確認する。品質が足りない場合は再生成する。
- 画像は本文の説明が十分に整ってから追加する。
- 画像は教材理解を助ける場合だけ追加する。雰囲気づくりだけの装飾画像は置かない。
- 各章には、学習者向けの文字入りインフォグラフィックを少なくとも1枚入れることを基本にする。
- 章内に画像を置かない判断をする場合は、既存画像の再掲で十分か、本文だけで迷わないか、ユーザーと合意できているかを確認する。
- 画像は数合わせで置かない。学習者が頭の中にモデルを作る必要がある場所、似た概念を取り違えやすい場所、操作順や判断基準を見失いやすい場所に置く。
- 画像は本文の直後に置くとは限らない。概念を先に固定したい場合は説明の早い段階に置き、操作後に意味を回収したい場合は「何が起きたのか」や「運用者の視点」で置く。
- 同じ画像でも、学習者が別の文脈で同じ概念を思い出す必要がある場合は再掲してよい。再掲は重複ではなく、理解と判断を固定するために使う。
- 画像を置く主な用途は、抽象概念の固定、似たものの比較、操作フローの整理、危険な判断の可視化、AIが見ている範囲や作業場所の確認です。
- 画像を追加する前に、第0部全体と第1部1〜4章の画像配置を正解例として見直し、本文と画像が同じ説明を別の角度から支えているか確認する。
- 汎用的なロードマップ画像、数字だけのステップ図、抽象アイコンだけの図は避ける。
- 章数や部構成は変わりやすいため、画像ファイル名に部番号・章番号・旧章番号を入れない。
- 画像ファイル名は内容ベースの英語 kebab-case にする。
  - よい例: `ai-partner.png`, `install-output-guide.png`, `macos-wsl-side-by-side.png`
  - 避ける例: `00-concept.png`, `01-start-here-roadmap.png`, `route-p0-03-setup.png`
- 画像は安定した用途ごとにディレクトリを分ける。
  - 第0部で使う画像: `docs/images/route-start/`
  - トップページや教材全体で使う画像: `docs/images/`
  - 今後の部で専用画像が増える場合は、章番号ではなく用途名のディレクトリを作る。
- 生成画像はプロジェクト内の `docs/images/` 配下に保存し、本文から相対パスで参照する。
- 既存の画像が内容に合う場合は重複生成しない。
- 画像を追加・移動・削除したら、Markdown参照とDocusaurusビルドを確認する。
- 使われていない画像は残さない。章構成変更で不要になった画像は削除する。
- 画像内に日本語テキストを入れる場合は、誤字、読めるサイズ、章内容との対応を確認する。
- 生成結果が教材内容と合わない場合や品質が低い場合は、無理に使わず削除または再生成する。
- altテキストは学習者向けの日本語で書く。

## Git方針

実際にcommitする場合は、小さく意味のまとまった単位にしてください。

まとまった構成変更、章ファイルの追加、数章分の本文追加、画像の追加や削除が終わったら、適切なタイミングでcommit/pushをユーザーに促してください。

commit前には、少なくとも次を確認してください。

- `git status --short`
- 差分の範囲が意図した作業に収まっていること
- Docusaurusビルドまたは必要な確認コマンドが通っていること
- 不要な画像、古い章ファイル、壊れたMarkdownリンクが残っていないこと

commitメッセージは、作業内容がわかる短いものにしてください。
例:

```text
docs: add part 1 chapter skeletons
docs: update route syllabus
docs: revise part 0 setup guide
```
