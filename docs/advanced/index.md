# 発展編

発展編は、基本編を終えたあとに、AIを自分の作業に合わせて育て、安全に使うためのカリキュラムです。

基本編では、AIと一緒に手を動かしながら、PC、CLI、Git、GitHub、Astro、GitHub Pages公開までを一本道で扱いました。
発展編では、その上に、AGENTS.md、プロンプトテンプレート、skills、安全確認、AIレビュー、サブエージェント運用を積み上げます。

## 発展編の考え方

発展編で扱う中心テーマは、AIに大きな作業を丸投げすることではありません。
AIが迷わず、安全に、確認しながら働ける作業環境を人間が設計することです。

ソフトウェア開発では、対象を動かしたり検証したりするための実行環境や仕組みを、ハーネスと呼ぶことがあります。
発展編で扱う作業環境づくりも、それに近い考え方です。
AIに見せる情報、使わせる道具、守らせる制約、確認手順を、人間が扱いやすい形に整えていきます。

その作業環境には、次のような要素が含まれます。

```text
AIに任せるための作業環境
  ├─ AGENTS.md
  ├─ 要件メモ
  ├─ 補助コンテキスト
  ├─ プロンプトテンプレート
  ├─ skills
  ├─ 安全チェック
  ├─ レビュー観点
  ├─ build / test / lint
  └─ サブエージェントの役割分担
```

AIに任せる範囲は広げます。
ただし、目的、判断、責任は人間が持ちます。

## 発展編でできるようになること

- リポジトリごとのAI作業方針を設計できる
- AGENTS.mdに、AIへ守ってほしいルールを書ける
- AGENTS.mdやCLAUDE.mdを肥大化させず、役割ごとに整理できる
- コンテキストウィンドウを意識して、重要な決定をファイルに残せる
- 要件メモ、設計メモ、公式ドキュメント、OSSリポジトリなどを補助コンテキストとして使える
- よく使う依頼をプロンプトテンプレートとして再利用できる
- skillsで、特定の作業に必要な知識や手順を切り出せる
- AIに実装させる前後で、安全確認と差分確認ができる
- 実装者、レビューアー、説明役など、AIの役割を分けられる
- サブエージェントを使う場合に、並列化してよい作業と危ない作業を分けられる
- 長期タスクを、小さなチェックポイントに分けて進められる
- 自分のプロジェクト用のAI開発環境を整えられる

## 発展編 第1部：AIに任せるための作業環境

### ゴール

AIに頼む内容だけでなく、AIが働く環境そのものを設計する視点を持ちます。

### 学ぶこと

- AIに任せるための作業環境とは何か
- AIへのお願いと、AIが働く仕組みの違い
- 指示、制約、道具、確認手順を分ける考え方
- AIに見せる情報の範囲を、人間が設計する考え方
- コンテキストウィンドウと、会話だけに頼らない作業設計
- AIに任せる範囲を広げる前に、確認できる構造を作る理由
- 人間が持つべき目的、判断、責任

### 到達目標

- AIに任せるための作業環境の役割を説明できる
- AIに任せる前に、何を決めるべきか整理できる
- 「AIを信じる」ではなく「確認できる形にする」と考えられる

## 発展編 第2部：AGENTS.mdを育てる

### ゴール

リポジトリごとの作業方針をAGENTS.mdに書き、AIの振る舞いを安定させます。
さらに、作業中に見つかったルールをAIと一緒にAGENTS.mdへ反映し、変更前後でAIの出力がどう変わるかを体験します。

### 学ぶこと

- AGENTS.mdは何を書く場所か
- 教材本文と、AIエージェント向け作業方針を分ける理由
- 文章の粒度、画像方針、問題方針、安全方針の書き方
- AIにやってほしくないことを禁止事項として書く方法
- AIにAGENTS.mdの改善案を出させる方法
- AIにAGENTS.md自体を編集させるときの確認手順
- 作業中に見つかったルールをAGENTS.mdへ反映する方法
- 同じ依頼でも、AGENTS.mdの有無や内容で出力が変わることを確認する実験
- AGENTS.mdを変更したあと、AIが新しい指示を読めているか確認する方法
- AGENTS.mdに書きすぎると読みにくくなる問題
- CLAUDE.mdやAGENTS.mdが肥大化したときの分割方針
- 全体方針、プロジェクト固有ルール、再利用する手順を分ける考え方

### 到達目標

- 自分のプロジェクトに合ったAGENTS.mdを書ける
- AIの失敗から、次に守るべきルールを抽出できる
- 本文に書くべきことと、AI向け指示に書くべきことを分けられる
- AGENTS.mdの変更前後で、AIの出力差を比較できる
- AGENTS.mdをAIに更新させる前に、変更予定と理由を確認できる
- AGENTS.mdに残すもの、リファレンスやskillsへ逃がすものを判断できる

## 発展編 第3部：コンテキストウィンドウと作業メモを理解する

### ゴール

長い会話や複雑な要件を、AIの記憶任せにせず、あとから読み直せる作業メモとして扱えるようになります。
また、AIに見せるコンテキストを、作業対象のリポジトリだけに閉じず、必要に応じて補助コンテキストまで広げる考え方を学びます。

### 学ぶこと

- コンテキストウィンドウとは何か
- 長い会話で、細かい条件が薄れたり要約されたりする理由
- セッションの新規開始、resume、compactの違い
- セッションを再開したとき、AIが何を覚えていて何を読み直すべきか
- 指示ファイルの読み込み直しが必要な場面と、その確認方法
- AIに質問してもらい、人間が回答する壁打ちの進め方
- AIとのやり取りを要件メモに起こす方法
- 目的、決まったこと、未決定のこと、やらないことを分ける方法
- 要件メモにも秘密情報を書かない判断
- 作業対象の外にある補助コンテキストを、AIに読ませる方法
- 要件メモ、設計メモ、公式ドキュメント、公式サンプル、OSSリポジトリなどの使い分け
- 作業対象と補助コンテキストを混同させないための指示
- 補助コンテキストは読み取り中心に扱い、勝手に編集させない方針
- compact後に、要件メモを読ませて作業を再開する方法
- 会話ログではなく、ファイルを作業の正本にする考え方

### 到達目標

- AIに質問役を頼み、要件を引き出してもらえる
- 壁打ちの結果をMarkdownの要件メモに整理できる
- 作業対象と補助コンテキストを区別してAIに渡せる
- resumeやcompactのあとに、AIへ読ませ直すべきファイルを判断できる
- compactや会話の切り替えがあっても、ファイルを読ませて作業を続けられる

## 発展編 第4部：プロンプトを作業テンプレートにする

### ゴール

毎回の依頼を場当たり的に書くのではなく、繰り返し使える作業テンプレートにします。

### 学ぶこと

- 良いプロンプトを、会話のコツではなく作業手順として扱う
- AIに質問役を頼むプロンプト
- 修正観点を洗い出すプロンプト
- レビューを依頼するプロンプト
- 問題を出してもらうプロンプト
- 画像作成を依頼するプロンプト
- commit前確認を依頼するプロンプト
- 一問一答、選択肢、採点タイミングなどの指定方法

### 到達目標

- よく使う依頼をテンプレート化できる
- AIに出してほしい形式を具体的に指定できる
- 使いにくかったAIの回答を、次のプロンプト改善につなげられる

## 発展編 第5部：skillsでAIをカスタマイズする

### ゴール

特定の作業で繰り返し使う知識や手順をskillsに切り出し、AIの作業品質を安定させます。

### 学ぶこと

- skillとは何か
- AGENTS.mdとskillの違い
- どんな作業をskillにするべきか
- ドキュメント編集skill
- 画像作成skill
- レビューskill
- 公開前確認skill
- skillに書くべきこと、書きすぎないほうがよいこと

### 到達目標

- AGENTS.mdに置く内容とskillに切り出す内容を分けられる
- 自分の作業に合った小さなskillを設計できる
- skillを増やす前に、本当に再利用する作業か判断できる

## 発展編 第6部：安全装置と確認手順を作る

### ゴール

AIの出力を信じ切るのではなく、build、test、lint、差分確認で安全に受け止める手順を作ります。

### 学ぶこと

- `git status` と `git diff` を確認の入口にする
- build、test、lint、リンクチェックの役割
- 危険コマンドと状態を変える操作の確認
- 秘密情報、`.env`、トークン、APIキーの確認
- 公開前チェックリスト
- 失敗した変更を立て直す考え方

### 到達目標

- AIの変更を受け入れる前に、確認すべき項目を説明できる
- 自分のプロジェクト用のチェックリストを作れる
- 危険な変更を、実行前または公開前に止められる

## 発展編 第7部：AIレビューを設計する

### ゴール

AIに実装させるだけでなく、複数の観点でレビューさせる流れを作ります。

### 学ぶこと

- 実装レビュー
- 初学者視点レビュー
- セキュリティレビュー
- 文章レビュー
- 画像と本文の対応レビュー
- 差分レビュー
- レビュー観点をAIに明示する方法
- レビュー結果をそのまま信じず、人間が採用判断する方法

### 到達目標

- AIにレビューしてほしい観点を分けて依頼できる
- レビューコメントを、対応するものと見送るものに分けられる
- 実装AIとレビューAIを分ける意味を説明できる

## 発展編 第8部：サブエージェントで役割分担する

### ゴール

サブエージェントを使える環境で、探索、実装、レビューを安全に分担します。

### 学ぶこと

- サブエージェントとは何か
- 並列化してよい作業と、並列化しないほうがよい作業
- 探索エージェントの使い方
- 実装エージェントの使い方
- レビューエージェントの使い方
- 書き込み範囲を分ける理由
- サブエージェントの結果を人間が統合する理由

### 到達目標

- サブエージェントに渡すタスクを具体的に切り出せる
- 複数のAI作業が衝突しないように範囲を分けられる
- 並列作業の結果を自分で確認し、採用判断できる

## 発展編 第9部：長期タスクをAIに任せる

### ゴール

AGENTS.md、要件メモ、プロンプト、skills、安全確認、レビュー設計を使い、長めの開発タスクを小さな確認単位で進めます。

### 学ぶこと

- AIと壁打ちして要件を作る
- AIに質問してもらい、回答を要件メモにまとめる
- 目的、制約、完了条件、やらないことを分ける
- 再帰型ToDoリストを作る
- チェックポイントを置く
- 調査、設計、実装、レビュー、検証を分ける
- 長期タスクが崩れたときに立て直す
- AIに任せた結果を人間が説明する

### 到達目標

- 長期タスクをAIに頼む前に、条件と確認方法を整理できる
- 大きな変更を小さな作業単位に分けられる
- 途中で方針がずれたときに、止めて立て直せる

## 発展編 第10部：自分専用のAI開発環境を作る

### ゴール

自分の成果物リポジトリに、AIと継続開発するための作業環境を整えます。

### 学ぶこと

- 自分のプロジェクト用AGENTS.mdを書く
- よく使うプロンプトテンプレートをまとめる
- 必要ならskillsを作る
- build、test、review、commit前確認の流れを整える
- AIに任せる範囲と、人間が確認する範囲を決める
- 今後の開発で育てていくルールを残す

### 到達目標

- 自分のプロジェクトに、AIと継続開発するための作業環境を導入できる
- AIと継続開発するための基本セットを説明できる
- AIに任せる量を増やしても、安全確認の軸を失わない
