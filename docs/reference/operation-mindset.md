# 運用者の視点

このページでは、作ったものを公開・維持する視点を整理します。

## 作って終わりではない

AIを使えば、以前より簡単に何かを作れます。

しかし、公開したり、誰かに使ってもらったりする場合、運用責任が生まれます。

## 維持コストとは

維持コストは、金銭だけではありません。

- 修正する時間
- 調査する時間
- エラー原因を理解する負荷
- セキュリティを確認する負荷
- ドキュメントを更新する負荷
- 利用者に説明する負荷
- 将来の自分が思い出すための負荷

## READMEの役割

READMEは、未来の自分や他の人に向けた説明です。

次のことが書いてあると、維持コストが下がります。

- これは何か
- どう使うか
- どう更新するか
- どこに公開されているか
- 注意点は何か

## Git履歴の役割

Gitの履歴は、問題が起きたときに原因を追うための記録です。

commitメッセージは、未来の自分へのメモでもあります。

## 公開前に考えること

- 何を公開しているか
- 誰が見られるか
- あとから修正できるか
- 秘密情報が含まれていないか
- AIが生成した内容を自分で説明できるか
- 維持できる規模になっているか

## 公開後の小さな運用

公開後に直したい点を見つけたら、慌てず小さく修正します。

```text
問題を確認する
↓
修正するファイルを決める
↓
git status と git diff を見る
↓
commitする
↓
pushする
↓
公開ページで反映を確認する
```

一度公開された内容は、誰かが見た可能性があります。秘密情報を公開した場合は、ページから消すだけでなく、キーの無効化や関係者への連絡が必要になる場合があります。

## AIに聞くプロンプト

```text
このリポジトリを公開・運用する場合、
初心者が見落としやすい維持コストと責任を整理してください。
技術面、セキュリティ面、ドキュメント面に分けて説明してください。
```
